# IRR-V1-00008 — Relatório do primeiro teste de campo

**Data:** 2026-04-30
**Dispositivo:** `IRR-V1-00008` (placa do protótipo, montagem 2026-04-28)
**Broker:** `mqtts://hub.xtconect.online:8883` (produção)
**UI:** `https://hub.xtconect.online` (branch `lovable-port`)

## Resumo

Primeiro teste ponta-a-ponta com a placa real conectando ao broker de produção. **Funcionou de primeira.** UI descobriu os 8 setores publicados pelo ESP, comandos remotos (XT Hub) e locais (botão na placa) operaram bomba e setores corretamente, telemetria periódica a cada 2s sem falha de publicação. Bugs encontrados são menores e não bloqueiam uso — listados em §3.

## 1. O que foi validado

### 1.1 Conexão e descoberta
- TLS + autenticação MQTT subiu de primeira (não houve rebanho de retries no log).
- UI carregou os 8 setores que o firmware publicou em `devices/IRR-V1-00008/state` (retain).
- `state` retain reflete corretamente cada transição (sector ON/OFF, pump ON/OFF, mode change).

### 1.2 Comandos remotos (XT Hub → ESP via MQTT)
| Comando         | cmd_id (sample)                       | Resultado |
|-----------------|---------------------------------------|-----------|
| `sector_open` 4 | `461dbfc8-…`                          | `executed/ok` |
| `sector_close` 4| `0fcf9077-…`                          | `executed/ok` |
| `pump_on`       | `ffc13a69-…`                          | `executed/ok` (com setor 4 aberto) |
| `pump_off`      | `02e3834d-…`                          | `executed/ok` |
| `mode_set` auto | `2080e188-…`                          | `executed/ok` |
| `set_rate`      | `bc4072a8-…`                          | `executed/ok`, telem voltou a 2s |
| `mode_set` man  | `6ec365d9-…`                          | `executed/ok` |

Todos seguidos de `MQTT PUB devices/IRR-V1-00008/state … retain=true -> OK` e o evento correspondente com `origem=manual_remote`.

### 1.3 Comandos locais (botão na placa)
- Setores 1, 2 e 3 abertos pela placa → estado refletido na UI imediatamente (`source=manual_local`).
- Bomba ligada localmente com setores abertos → `pump_on_manual / origem=manual_local`.

### 1.4 Telemetria
- `devices/IRR-V1-00008/data` publicado a cada 2s, payload ~115-116 B.
- Janela observada: 2s estável durante toda a sessão (~3 min de log capturado).

### 1.5 Proteção bomba seca (UI)
- Tentativa de ligar bomba pela UI com todos setores fechados → modal de aviso (sem PUB de comando ao broker).
- Confirmado no log: nenhum `MQTT RX commands` para `pump_on` enquanto setores estavam todos fechados.

## 2. Pontos certos (vale registrar)

- **Idempotência por `cmd_id`** funcionou — não houve replay/duplicação observada.
- **State retain consistente** — cada mudança gera PUB com retain=true, o que permite UI nova abrir e ter snapshot.
- **`set_rate` com `cmd_id` e ACK** — implementado conforme protocolo v2 (era débito da spec).
- **Mudança de modo automatico ↔ manual via comando remoto** — confirmado, com echo em `config/current` retain.

## 3. Pontos errados / bugs encontrados

### B1. Timestamps `[NaN:NaN]` no log antes do SNTP
Toda linha do log (`/log` na AP) aparece com prefixo `[NaN:NaN]`. O uptime `mm:ss.ms` está correto. Provavelmente formatador chama `localtime(time(NULL))` antes de SNTP fechar. **Cosmético**, mas atrapalha correlacionar log local com timeline servidor.

### B2. `source=` vazio em desligamentos em cascata
Eventos secundários disparados por safety/cascata saem com `source=` vazio:

```
01:35.7 I SETOR 3 OFF source=
02:10.3 I BOMBA OFF source=
02:11.0 I SETOR 1 OFF source=
```

Afeta auditoria — Influx fica com `source` nulo nesses eventos.

### B3. Linhas duplicadas/triplicadas na página `/log`
Trecho do final do log capturado:

```
02:50.2 I TELEM PUB ... -> OK
02:52.2 I TELEM PUB ... -> OK
02:50.2 I TELEM PUB ... -> OK   ← repete
02:52.2 I TELEM PUB ... -> OK   ← repete
02:54.2 I TELEM PUB ... -> OK
02:56.2 I TELEM PUB ... -> OK
02:54.2 I TELEM PUB ... -> OK   ← repete
02:56.2 I TELEM PUB ... -> OK   ← repete
02:54.2 I TELEM PUB ... -> OK   ← terceira vez
```

Provavelmente render concorrente do ring-buffer + wraparound, sem snapshot atômico ou dedup por seq.

### B4. `requires_confirmation` (spec §6.2) ainda não implementado no firmware
A spec prevê: firmware recebe `pump_on` sem `force`, sem nenhum setor aberto → responde `ack_status=requires_confirmation`. UI então oferece "Ligar mesmo assim" e reenvia com `force=true`.

Hoje a UI bloqueia totalmente no front (`beforePumpOn` retorna false). Funciona pro caso normal, mas não há como forçar pra teste/manutenção sem entrar no modo automatico/local. Documentado em FIRMWARE-QUICKSTART.md §B4.

## 4. O que mudou no front pós-teste

- `IrrigacaoDashboardPage.tsx` — modal "Nenhum setor aberto" passou de texto curto + botão "Entendi" pra:
  - Título: "Abra um setor antes de ligar a bomba"
  - Descrição mais explicativa (causa do dano + caminho de ação).
  - Botão: "Ok, vou abrir um setor".
- Mesmo arquivo — adicionada dica visual ambar abaixo do botão da bomba quando ela está OFF e nenhum setor aberto, sem precisar clicar pra ver o aviso.

## 5. Próximos passos

### No firmware (próximo ciclo)
1. **B1** — formatar `[--:--]` antes de SNTP, e bloquear PUB com `ts` até hora válida (já é regra).
2. **B2** — propagar `source` original em cascatas (sector→pump, pump→sector safety).
3. **B3** — snapshot atômico do ring buffer + dedup por seq na página `/log`.
4. **B4** — implementar `requires_confirmation` pro `pump_on` sem setor (e aceitar `force=true`).

### No front (segundo passo, depois do firmware suportar `force`)
- Botão "Ligar mesmo assim (manutenção)" no modal `pump_on_without_sector`, com aviso vermelho, que reenvia `pump_on {force:true}`.
- Tratar `ack_status=requires_confirmation` retornado pelo firmware (`useComando`/`decisionOpts`) — abrir o mesmo modal com a opção de forçar.

### Operacional
- Manter dispositivo em uso por algumas horas pra validar reconnect após queda Wi-Fi e idempotência de comandos retransmitidos.
- Conferir Influx — todos eventos da sessão devem estar em `device_events` com `device_id` correto.
