## O que está rodando

Site no ar em <https://hub.xtconect.online/> ✓

5 serviços Docker, todos com selo "saudável":

- **Banco de dados** (Postgres) — guarda os dados
- **Mosquitto** — comunicação MQTT com os equipamentos
- **Node-RED** — lógica de negócio (regras, alarmes, notificações)
- **InfluxDB** — gráficos de telemetria
- **Nginx** — recebe as requisições do navegador

Espaço em disco: usando **6.9 GB de 193 GB** (4%). Sobra 186 GB pro futuro.

Tabela mais pesada do banco: `device_readings` com 1.1 MB. Sem risco de banco encher tão cedo. Ainda assim, retenção automática roda 03:00 todo dia.

## O que o sistema faz hoje

### Pra você como administrador

1. Cadastrar produtos (modelos), gerar códigos de associação, regenerar senhas MQTT
2. Ver lista de todos os clientes e seus dispositivos
3. Backup do banco automático todo dia às 03:00, mantém 7 cópias diárias + 30 semanais + 365 mensais

### Pra qualquer usuário cadastrado

1. **Adicionar equipamento** com código de pareamento ou QR
2. **Painel de irrigação completo**: bomba, setores, sensores de temperatura (com gauge analógico + digital), timers (3 tipos: fixo, cíclico janela, cíclico contínuo), histórico, logs, configuração de hardware
3. **Compartilhar equipamento** com outro usuário (visualizar ou comandar)
4. **Reconhecer alarme** — sistema mostra mensagem destacada, botão Reconhecer
5. **Criar automações**: "se acontecer X, manda email pra fulano e/ou desliga bomba"
6. **Receber e-mails automaticamente**:
   - Alarme de temperatura disparou
   - Equipamento ficou offline > 30 min (formato "5 dias 22:56H" pra fácil leitura)
   - Quando alguém compartilha equipamento contigo
   - Verificação de cadastro

### Funcionando em modo seguro

- Alarme dispara cortando bomba e setores na hora (firmware/simulador, não depende do servidor estar online)
- Alarme só limpa quando alguém clica "Reconhecer" no app **ou** no simulador
- Se o servidor cair, os equipamentos continuam funcionando localmente

## Pendências e decisões

### Importante quando começar com clientes pagantes

1. **Trocar a senha do `/red/`** (combinado com o dono).
2. **Adicionar IP allowlist no `/red/`** — quando IP fixo do escritório for definido, fechar a porta. Snippet pronto em `infra/nginx/default.conf` (instruções no commit `c0ec3b7`).
3. **Push pro GitHub** — repo novo ainda não foi criado. 50 commits só localmente. Risco baixo (já tem backup do banco) mas vale criar repo. Push como `xtadmin`:
   ```bash
   sudo -u xtadmin git -C /opt/xtconect/src/webapp push -u origin lovable-port
   ```

### Postergado por decisão do dono

4. **Multi-empresa (Plano D)** — só faz sentido quando aparecer cliente B2B querendo XT Connect como serviço pra usar com clientes próprios.
5. **WhatsApp (Plano F)** — dono falou "futuramente te chamo".

### Externo ao código

6. **Firmware ESP real** — hoje só o simulador conversa com o servidor. Quando o firmware físico estiver pronto, conversa com o mesmo backend (protocolo em `infra/docs/`).

### Pendências técnicas pequenas (sem urgência)

7. Bug `set_rate` legado — corrigi parte do servidor (E.2), handler antigo ainda existe.
8. D-token-1: ACLs antigas em roles dynsec não são limpas imediatamente quando se revoga compartilhamento de dispositivo. Vão até o próximo POST `/api/mqtt-token` do mesmo user.
9. D-acl-1: isolamento por convention (`username = serial`), não por ACL literal. Hardening real seria role-per-device ou TLS client-cert.
